# workflows/compile.md — Compilation, ATS Check and Archiving

Post-processing for applications generated by `workflows/apply-batch.md`.
Mechanical — doesn't touch the tracker (Notion/CSV/none). Optionally takes a
date (`YYYY-MM-DD`); defaults to today if omitted.

**This step never registers the application anywhere.** The tracker is only
touched in `workflows/confirm.md`, after the dev confirms it was actually
sent — this avoids spending an API call (Notion) or a CSV row on an
application that might not even become a real submission.

## 1. Compile, ATS check and archive (delegate everything to the script)

```bash
python3 scripts/validate_profile.py   # fail fast if the profile is broken
bash scripts/compile-all.sh <date>
```

This one script does the entire mechanical pipeline — **do not stop after
running it and do not try to do any of the following by hand**:

1. Compiles each `.tex` referenced in `daily/<date>/*.json` (via `jq`, never
   guessing the filename).
2. Checks the page count is exactly 1.
3. Runs the ATS check via `grep` (see `rules/ats-verification.md`) — email
   and phone from the profile, stack/gap keyword coverage.
4. **Archives** every job that compiled to exactly 1 page and has no
   blocking ATS issue: moves the PDF and `.tex` into
   `documents/applications/<company>_<role>/` (as `main_<company>.pdf` and
   `cv_draft.tex`), copies the day's JSON there as `metadata.json`, and
   writes `outcome.md` with status "waiting for send confirmation".

Archiving used to be a manual step described here for whichever agent ran
this workflow — it's folded into the script now because a weaker model
reliably ran the compile step and then stopped, silently skipping the
archive (see `rules/README.md` → Robustness for free/weaker models). If
you're an agent reading this: there is no step where you personally move
files or write `outcome.md` — the script already did it. Your only jobs are
step 2 (handle failures) and step 3 (report).

## 2. Handle failures

The script itself tells you, per job, whether it archived or not:

- `📦 Archived to ...` — done, nothing else to do for this job.
- `⚠ Not archived — needs manual review` — either the page count wasn't 1,
  or an `absent`-status gap literally appeared in the compiled text (a sign
  of invented data). Look at the log lines just above for which one.
  - Wrong page count: trim content per `rules/cv-rules.md`, fix the `.tex`
    in `documents/cv/` (the script only moves it on success, so it's still
    there), and rerun `bash scripts/compile-all.sh <date>`.
  - Invented-looking gap: fix the `.tex` to remove the offending claim,
    then rerun.
  - "full/partial gap missing" warnings are **not** blocking by themselves
    (may be a synonym false positive — see `rules/ats-verification.md`) and
    don't prevent archiving.
- `❌ COMPILE ERROR` — read the log printed above (usually an unescaped
  character or overfull hbox), fix the `.tex`, and rerun once. If it fails
  again, mark it as an error in the final report and move on.

## 3. Final report

Include the URL for every archived job (read it back from
`documents/applications/<slug>/metadata.json` if needed):

```
✓ Company X → 1pg ✅ ATS OK ✅ archived → <url>
✗ Company Y → COMPILE ERROR (overfull) → needs manual intervention
8/10 CVs compiled and archived. Go submit at the URLs above, then run
workflows/confirm.md — it's the only step that writes to the tracker.
```
